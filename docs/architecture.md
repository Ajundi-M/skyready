# SkyReady — Architecture

SkyReady is a **vigilance-training web app** built with Next.js 16 (App Router), Supabase, and TypeScript. Users earn access through invite codes, complete timed vigilance sessions, and track their performance on a personal dashboard. Admins manage users and invite codes through a protected panel.

## Auth Flow

Access to SkyReady is gated by invite codes. Signup requires a valid, unclaimed code; there is no open registration.

```
┌─────────────────────────────────────────────────────────────────────┐
│  SIGNUP                                                             │
│                                                                     │
│  User fills form                                                    │
│    email · password · display name · invite code                   │
│           │                                                         │
│           ▼                                                         │
│  Step 1 — Pre-flight check                                          │
│    SELECT FROM invite_codes                                         │
│    WHERE code = $1 AND used = false AND expires_at > now()          │
│    → invalid / expired → redirect /signup?error=...                │
│           │                                                         │
│           ▼                                                         │
│  Step 2 — Create Supabase Auth user                                 │
│    auth.signUp({ email, password, emailRedirectTo })                │
│    → always redirect to /signup/confirm (avoids email enumeration)  │
│           │                                                         │
│           ▼                                                         │
│  Step 3 — Atomic claim via SECURITY DEFINER RPC                    │
│    claim_invite_code(p_code, p_user_id)                             │
│    → single conditional UPDATE, eliminates TOCTOU race              │
│    → if claimed = false → auth.admin.deleteUser() + redirect error  │
│           │                                                         │
│           ▼                                                         │
│  User clicks confirmation email → /auth/callback                   │
│    exchangeCodeForSession() → session cookie                        │
│    → redirect /dashboard                                            │
└─────────────────────────────────────────────────────────────────────┘
```

**Login** is a standard email/password flow via `supabase.auth.signInWithPassword()` in a Server Action.

**Session management** uses the `@supabase/ssr` cookie adapter. `createClient()` (in `lib/supabase/server.ts`) reads/writes the session cookie on every server-side request.

**Admin check** — `requireAdmin()` (`lib/auth/require-admin.ts`) verifies the session user has `profiles.is_admin = true`. It returns `{ user, supabase }` on success or a `Response` (401/403) on failure, and is applied to every admin API route handler.

## Invite Flow

Invite codes are single-use tokens with a 48-hour expiry, generated by admins.

```
┌──────────────────────────────────────────────────────────────────┐
│  ADMIN creates code                                              │
│    POST /api/admin/invites                                       │
│    → generateInviteCode()  →  XXXX-XXXX-XXXX                   │
│       (32-char alphabet, excludes I/O/0/1 for visual clarity)   │
│    → inviteExpiry()        →  Date.now() + 48 h                 │
│    → INSERT invite_codes (code, expires_at, created_by)         │
│           │                                                      │
│           ▼                                                      │
│  Admin copies code and shares it out-of-band                    │
│           │                                                      │
│           ▼                                                      │
│  User enters code at signup (see Auth Flow above)               │
│           │                                                      │
│           ▼                                                      │
│  claim_invite_code() marks code:                                │
│    used = true · used_by = user_id · used_at = now()            │
└──────────────────────────────────────────────────────────────────┘
```

- Admin can also delete unused codes via `DELETE /api/admin/invites/[code]`.
- All admin routes are protected by `requireAdmin()` and a same-origin CSRF check (`checkCsrfOrigin()`).
- Admin invite POST is additionally rate-limited (20 requests per hour per IP).

## Session Lifecycle

A session represents one completed (or abandoned) vigilance training run.

```
                      POST /api/sessions
                      { user_id, module: 'vigilance' }
                               │
                               ▼
                         ┌─────────┐
                         │ Started │  sessions row created,
                         │         │  completed_at = NULL
                         └────┬────┘
                              │
              ┌───────────────┼───────────────────┐
              │               │                   │
              ▼               ▼                   ▼
         Completes       Navigates away      Tab hidden /
         normally        mid-session         focus lost
              │               │                   │
              ▼               ▼               (game paused,
    PATCH /api/sessions/:id   │             scoring stops)
    { duration_s, score,      │
      accuracy, metrics,      │  onCancel fires → PATCH with
      completed_at }          │  metrics.cancelled = true
              │               │
              ▼               ▼
         ┌──────────┐   ┌──────────┐
         │Completed │   │Cancelled │
         └──────────┘   └──────────┘

    Sessions with completed_at IS NULL
    and started_at < now() - 1 hour
    are pruned by a scheduled SQL job
    (see supabase/migrations/…cleanup_abandoned_sessions.sql)
```

The PATCH body is validated server-side with `patchSessionSchema` (Zod), which enforces:

- `duration_s` ∈ [0, 3600]
- `accuracy` ∈ [0, 100]
- `score` ∈ [−duration_s × 2, duration_s × 2] (plausibility bound)
- No unknown keys (strict mode)

## Data Model

```
auth.users  (managed by Supabase Auth)
    id (uuid) PK
    email
    ...

profiles
    id (uuid) PK → auth.users.id
    email
    display_name
    is_admin      boolean  default false
    created_at

invite_codes
    id (uuid) PK
    code          text  UNIQUE  — format: XXXX-XXXX-XXXX
    created_by    uuid → profiles.id
    expires_at    timestamptz
    used          boolean  default false
    used_by       uuid → profiles.id  (nullable)
    used_at       timestamptz         (nullable)
    created_at

sessions
    id            uuid PK
    user_id       uuid → profiles.id
    module        text  — currently always 'vigilance'
    started_at    timestamptz
    completed_at  timestamptz  (nullable until PATCH)
    duration_s    integer
    score         integer
    accuracy      numeric
    metrics       jsonb  — see SessionMetrics type in lib/schemas/session.ts
```

**Row-Level Security** is enabled on all tables. The service-role client (`createServiceClient()`) is only used where the authenticated role lacks permission:

- `claim_invite_code` RPC — needs to update `invite_codes` before the user has an active session cookie.
- `auth.admin.deleteUser()` — only available to the service role.
- Admin GET `/api/admin/invites` and `/api/admin/users` — fetch rows that RLS restricts to their owner.

## Security Model

| Concern             | Mechanism                                                                                                                |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| Input validation    | Zod schemas on all mutating API routes                                                                                   |
| CSRF                | Same-origin `Origin`/`Referer` check on admin mutating routes                                                            |
| Rate limiting       | In-memory sliding-window limiter on login, signup, admin invite POST                                                     |
| Admin authorisation | `requireAdmin()` helper on every admin route                                                                             |
| Security headers    | `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy` via `next.config.ts` |
| Service role        | Used only where necessary; each call site is documented with a comment explaining why                                    |

> **Note:** The in-memory rate limiter resets on cold starts (serverless deployments spin up fresh instances). Redis/Upstash would be needed for distributed rate limiting in production.
